### 차이 #1: bare except vs except Exception [임의 판단 — 코드 스타일]
- 원본: `except:` → KeyboardInterrupt, SystemExit 도 catch
- 우리: `except Exception:` → KeyboardInterrupt, SystemExit 통과
- **영향**: GrabCut 실행 중 Ctrl+C를 누르면 원본은 무시하고 계속, 우리는 종료.
  실질적으로 GrabCut이 이 예외를 발생시키지 않으므로 **결과 차이 없음**.

### 차이 #2: bbox_area == 0 체크 [임의 판단 — defensive coding]
- 원본: bbox_area가 0이면 `mask_proportion` 계산에서 division by zero 발생 가능
- 우리: 명시적으로 체크하여 `None, False` 반환
- **영향**: 실제로 ImageNet bbox에 area=0인 케이스는 없으므로 **결과 차이 없음**.
  단, edge case 안전성 향상.

### ~~버그~~ (수정 완료): FRACTION check에서 train용 transform 사용

**수정 전** (bg_challenge/generate_in9.py에 아직 남아있음):
```python
# 잘못된 코드 (train이면 256 crop 사용)
crop_size = TEST_CROP if split == "val" else TRAIN_CROP
xform = test_transform if split == "val" else standard_transform
```

**원본 코드 (line 167)**:
```python
post_crop = test_transform(mask)  # 항상 224 crop
```

**수정 후** (in9-dataset-generator/generate_in9.py):
```python
post_crop = test_transform(mask)  # 항상 224 crop (원본과 동일)
```

**영향**: 수정 전에는 train에서 256 crop을 사용하여 FRACTION 기준이 더 관대했음.
224 crop은 이미지 가장자리를 더 많이 잘라내므로 bbox가 차지하는 비율이 더 높아지고,
따라서 더 많은 이미지가 FRACTION>0.9 기준으로 필터링됨.
**결론**: 수정 전에는 원본보다 더 적은 이미지가 필터링 탈락 → 약간 더 많은 이미지 통과.

**왜 원본이 항상 224를 쓰는가**: 최종 평가(test)는 CenterCrop(224)를 사용하므로,
"crop 후 bbox 비율"을 평가 시점 기준으로 체크해야 일관성이 있다.
Train 이미지를 256으로 저장하더라도, 이 필터링은 "평가할 때 쓸모 있는 이미지인가"를 판단하는 것.

### 차이 #3: 파일 정렬 [임의 판단 — 재현성]
- 원본: `os.listdir()` → OS 의존적 순서 (비결정론적)
- 우리: `sorted()` → 알파벳 순 (결정론적)
- **영향**: 처리 순서만 다르고 최종 결과에는 영향 없음. 다만 로그 출력 순서가 다를 수 있음.

### 차이 #4: 파일 확장자 필터링 [임의 판단 — defensive coding]
- 원본: 모든 파일을 처리 시도 (ImageNet 디렉토리에 .JPEG만 있다고 가정)
- 우리: `.jpeg`, `.jpg`, `.png` 확장자만 처리
- **영향**: ImageNet 디렉토리에 비이미지 파일 (e.g., .DS_Store)이 있으면 원본은 에러,
  우리는 무시. **결과 차이 없음** (정상적인 ImageNet에는 .JPEG만 존재).

### 차이 #5: 파일 이름 형식 [임의 판단 — 코드 스타일]
- 원본: `{synset}_{image_num}.JPEG` (e.g., `n02279972_01234.JPEG`)
  - `image_num`은 `im_file[-10:-5]`로 추출한 5자리 (val: `01234`)
- 우리: 원본 파일명 그대로 (e.g., val: `ILSVRC2012_val_00001234.JPEG`, train: `n02279972_1234.JPEG`)
- **영향**: 파일 이름만 다르고 이미지 내용은 동일. 평가 코드(ImageFolder)는 파일명을 사용하지 않으므로
  **결과 차이 없음**.
- Mixed 파일명도 형식이 다름:
  - 원본: `fg_n02279972_01234_bg_n02279972_56789.JPEG`
  - 우리: `fg_ILSVRC2012_val_00001234_bg_ILSVRC2012_val_00005678.JPEG` (val) 또는
    `fg_n02279972_1234_bg_n02279972_5678.JPEG` (train)

### 차이 #6: 디렉토리 구조 — synset vs superclass index [의도적 변경 — 기능 확장]
- 원본: `{out_dir}/{variant}/val/{synset}/` (e.g., `output/original/val/n02279972/`)
- 우리: `{out_dir}/{variant}/{split}/{sc_index}/` (e.g., `output/original/train/5/`)
- **이유**: 원본은 단일 synset 처리이므로 synset별 폴더가 자연스러움.
  우리는 9 superclass 전체 처리이고, 평가 코드 (`in9_eval.py`의 `ImageNet9` 클래스)가
  superclass index (0-8) 폴더를 기대하므로 평가와 직접 호환되는 구조 채택.
- **영향**: 평가 시 별도 변환 불필요. 단, 원본 출력과 직접 비교는 불가 (폴더 구조 상이).

### 차이 #7: only_bg_t 실패 시 fallback [임의 판단 — defensive coding]
- 원본: `get_bg_tiled()`가 `ValueError`를 raise하면 **프로그램 크래시** (또는 assert 실패)
  - 실제로 bbox가 이미지 전체를 차지하면 배경 tile 영역이 없어 발생 가능
  - 그러나 FRACTION check (step 5)에서 bbox > 90%면 이미 필터링되므로,
    `get_bg_tiled()` 실패가 발생할 확률은 극히 낮음
- 우리: `try/except ValueError` → `bg_t = bg_b` (blackout으로 fallback)
- **영향**: 거의 발생하지 않는 edge case. 원본은 크래시, 우리는 graceful degradation.
  단, 이론적으로 only_bg_t 결과가 달라질 수 있음 (blackout vs tile).

### 차이 #8: only_fg/no_fg 생성 시점 + fg_mask 저장 [의도적 변경 — 품질 개선]
- 원본:
  - Phase 1: original, bg_b, bg_t만 저장 (GrabCut 성공해야)
  - Phase 2: GrabCut **재실행** (최대 10회 retry) → only_fg, no_fg, mixed 생성
  - 이유: "Need to redo segmentations from scratch to avoid JPEG compression issues" (line 297-298)
    Phase 1에서 only_fg를 JPEG로 저장 → Phase 2에서 로드하면 pixel==[0,0,0] 체크 시 JPEG 아티팩트로 오류
- 우리:
  - Phase 1: 모든 비합성 variant + fg_mask.npy 저장
  - Phase 2: .npy에서 mask 로드 → only_fg 재생성 → mixed 합성
  - 이유: .npy는 lossless이므로 JPEG 문제 없음. GrabCut 재실행 불필요.
- **영향**: SAM (deterministic)에서는 차이 없음.
  GrabCut (non-deterministic)에서는:
  - 원본: Phase 2에서 다른 mask가 나올 수 있음 (retry로 "더 나은" mask 기대)
  - 우리: Phase 1의 mask를 그대로 사용 (retry 없음)
  - **GrabCut 모드에서 only_fg/no_fg/mixed 결과가 원본과 다를 수 있음**.

### 차이 #9: BG 선택 범위 — synset vs superclass [의도적 변경 — 논문 정의 준수]
- 원본: `get_rand_image(f"{out_dir}/original/val/{synset}")` → 같은 **synset** 내에서 BG 선택
  - 이유: 원본 코드가 단일 synset 단위로 동작하는 구조적 한계 (다른 synset 데이터 없음)
  - 논문 Section 3: "same class"라고 기술 → IN-9의 "class"는 **superclass**
- 우리: `all_good_images[bg_sc]` → 같은 **superclass** 전체 synset에서 BG 선택
  - 예: SC 0 (Dog, 116 synsets) → 116개 synset 모두의 good_images 풀에서 선택
- **영향**:
  - 원본 (synset 범위): mixed_same BG가 FG와 매우 유사한 종 (e.g., 같은 개 품종)
  - 우리 (superclass 범위): mixed_same BG가 같은 대분류지만 다른 종 (e.g., 다른 개 품종)
  - BG-Gap 측정에 영향 가능: BG가 FG와 더 유사할수록 "hint" 효과 강화 → BG-Gap 증가

### 차이 #10: mixed_rand, mixed_next 구현 [의도적 변경 — 기능 확장]

- **영향**: 신규 기능이므로 원본과 비교 대상 아님. BG-Gap 측정에 필수.

## 9. only_fg, no_fg 저장 (원본 line 356-358, 402-406)

## 14. 결론

### 결과에 실질적 영향을 주는 차이점: **2가지**만

1. **#9 — BG 선택 범위 (synset vs superclass)**
   - 원본: 같은 synset (구현 한계)
   - 우리: 같은 superclass (논문 정의)
   - 이것은 의도적인 변경이며, 논문의 "same class" 정의를 따른 것.
   - BG-Gap에 약간의 영향 가능하나, 논문 범위(6-22%) 내일 것으로 예상.

2. **#8 — GrabCut retry 없음 (GrabCut 모드 전용)**
   - SAM 3 모드에서는 영향 없음 (deterministic).
   - GrabCut 모드에서는: Phase 2의 mask가 Phase 1과 동일 (retry 없음).
   - 원본은 Phase 2에서 재실행 + retry → 다른 mask 가능.
   - **우리의 주 사용 segmentor는 SAM 3이므로 실질적 영향 없음**.

### 나머지 16개 차이점
- 임의 판단 (defensive coding, 재현성, 스타일): **결과 영향 없음**
- 의도적 변경 (기능 확장): **원본에 없는 신규 기능**

### "원본 재현"을 위한 체크리스트

GrabCut 모드 (`--segmentor grabcut`)로 val set을 단일 synset 처리하면,
차이 #9 (BG 범위)를 제외하고 원본과 동일한 결과를 재현할 수 있다.
BG 범위 차이도 단일 synset 모드에서는 해당 synset만 있으므로 사실상 동일.
